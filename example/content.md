# 起點：為什麼不要停在一次性生成
> 這門課從一份講稿出發，示範如何把 AI 產生的內容整理成可維護的課程頁，而不是只得到一個難以修改的 HTML。

## 從 PPT 走向網頁

### Web 比投影片更靈活
- 課程內容可以在桌機、平板、手機上閱讀，也能再匯出成 PDF。
- 版型與互動效果能重複使用，不需要每次都從頭排版。
- 修改時只要編輯 Markdown，不必反覆碰脆弱的 HTML 結構。

### 一次性輸出的問題
- 直接讓 AI 產生完整網頁，通常第一版很快，但細節很難穩定維護。
- 只靠對話微調，容易改壞別的地方，結果每次都不一樣。
- 如果沒有模板與規範，下次做新課又得重來一次。

> **重點不是只做出來**
> 真正的價值在於把輸出格式、內容結構與 build 流程固定下來，之後每次都能沿用。

[tags]
- [green] 可重複
- [blue] 可維護
- [orange] 避免一次性
[/tags]

# 工具：先把本機工作流跑通
> 課程先示範如何把 Codex 與本地環境接起來，讓 AI 不只是聊天，而是真的能幫你建立專案與產生頁面。

## 安裝與登入

### 準備 Node.js
- 這個專案需要 Node.js 才能執行 build script 與本機預覽。
- 如果還沒有安裝，可以先用 `nvm install --lts` 建立穩定版本環境。

```prompt [label="檢查 Node.js"]
node -v
```

### 安裝 Codex
- 安裝完成後，就能在終端機裡直接用 Codex 協助建檔、轉檔與啟動專案。
- 課程也示範了用 ChatGPT 帳號或 API Key 登入的做法。

```prompt [label="安裝 Codex"]
npm i -g @openai/codex
```

### 在編輯器中協作
- 不管是 VS Code、Cursor 或其他編輯器，重點都是讓 AI 能在專案資料夾裡工作。
- 與其把 AI 當成一次性生成器，不如把它當成會操作專案的協作夥伴。

> **空資料夾也是起點**
> 只要工作目錄清楚、依賴裝好，後面就能把內容、模板與自動化流程逐步補齊。

# 模板：把內容與呈現拆開
> 這一步的核心是把課程內容改成結構化 Markdown，讓 HTML 成為 build 的結果，而不是手工維護的主體。

## 用 Markdown 管理內容

### 為什麼要先抽象
- 如果直接維護 HTML，內容、樣式與互動會混在一起，修改成本很高。
- 改成 Markdown 後，可以先專注在課程結構，再交給腳本轉成網頁。
- 這樣也比較容易讓 AI 理解「哪些是章節、哪些是重點、哪些是指令」。

### 讓 AI 幫你設計格式
- 可以直接要求 AI 先把課程頁拆成大標、中標、重點卡片、流程與摘要。
- 接著補一支 build script，讓 Markdown 更新時 HTML 自動重建。
- 這個 repo 的 skill 就是在做這件事。

```prompt [label="啟動範例課程預覽"]
npm run dev:example
```

## Build 與預覽

[flow]
1. 準備 `content.md` 與 `config.yaml`
2. 用 `build.mjs` 產生 `index.html`
3. 用 `generate-og.mjs` 輸出社群縮圖
4. 用 `dev.mjs` 啟動預覽並監看檔案更新
[/flow]

### 預覽比手改 HTML 更穩定
- 內容調整時只要回到 Markdown，不需要在大量 HTML 中找段落。
- 預覽伺服器會在存檔後自動重建，讓修正節奏更流暢。

> **這裡的關鍵是分層**
> Markdown 負責內容，config 負責課程設定，腳本負責把它們組合成頁面。

# Skill：把流程封裝成可重複使用的能力
> 當模板可以穩定產出後，下一步就是把這套做法變成 Agent Skill，讓之後的新主題也能沿用同一條路徑。

## 什麼是 Agent Skill

### Skill 不只是提示詞
- 它包含觸發描述、工作步驟、參考文件與腳本位置。
- AI 讀到需求後，不用每次都重新猜流程，而是照著既定工作流執行。
- 這樣做能大幅減少每次生成結果風格不一致的問題。

### Skill 檔案裡放什麼
- `SKILL.md`：定義何時觸發、怎麼轉換、怎麼 build。
- `reference/`：放格式範例、元件說明與設定樣板。
- `scripts/`：放 build、dev、OG 產圖等可直接執行的工具。

## 讓 AI 依流程工作

### 從主題或講稿都能開始
- 使用者只給主題時，Skill 可以先建立課程資料夾與骨架內容。
- 使用者已經有講稿時，Skill 則把原稿重新整理成符合模板的 `content.md`。
- 完成後再接續 build、OG 與預覽，不需要人工拆很多步。

### 可複用的價值
- 一旦工作流固定，新課程就只是換內容，而不是重做整個網站。
- 模板會越用越穩，Skill 也會跟著實戰經驗一起進化。

> **把經驗變成系統**
> 最值得保留的不是某一次輸出，而是那套之後還能持續複用的製作方法。

[summary]
- `工具層` | 安裝 Node.js 與 Codex，先讓本機工作流能運作
- `內容層` | 用 Markdown 描述章節、重點、流程與摘要
- `系統層` | 用 Skill、reference 與 scripts 封裝產出流程
[/summary]

# 部署：讓課程頁真的能被分享
> 當頁面產出完成，最後一段就是把它放上 GitHub Pages，並補齊 SEO 與 Open Graph，讓分享時有正確預覽。

## 上線與分享

### GitHub Pages 足夠實用
- 這類靜態課程頁不需要複雜後端，推上 GitHub 就能部署。
- 如果頁面在子目錄下，分享時要帶上對應路徑，例如這個專案的 `example/`。

### OG 與 SEO 不能少
- 如果沒設定 title、description 與 image，貼到社群平台時常常只會出現不完整預覽。
- 透過 `generate-og.mjs` 產生固定尺寸縮圖後，分享品質會穩定很多。

```prompt [label="產生範例頁面與 OG 圖"]
npm run build:example
npm run og:example
```

### 自由度越高越要收斂
- 網頁模板看起來自由，但真正上課、列印、分享時，細節問題才會浮現。
- 例如側欄占比、投影可讀性、列印版面與深淺色主題，都需要實際使用後調整。
- 所以模板不是做完就結束，而是需要被持續優化。

> **Vibe Coding 的下一步**
> 不是只追求生成速度，而是用系統設計把 AI 輸出變成真正可長期使用的工具。

[summary]
- `生成` | 先快速做出第一版課程頁
- `收斂` | 用 Markdown 與模板穩定內容格式
- `封裝` | 用 Agent Skill 固定流程與產出方式
- `部署` | 發佈到 GitHub Pages，讓成果能真正被分享
[/summary]
